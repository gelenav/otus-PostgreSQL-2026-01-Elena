## Выбор таблицы для секционирования

1. tickets: 21 095 265 записей, т.е. данных много, но отсутствуют поля, которые можно привязать к диапазонам для секционирования. 
2. flights: 135 571 записей - нет необходимости в секционировании
3. seats: 1741 записей - нет необходимости в секционировании
4. airports_data: 5501 записей - нет необходимости в секционировании
5. airplanes_data: 10 записей - нет необходимости в секционировании
6. boarding_passes: 26 299 160 записей, можно секционировать по времени посадки (boarding_time) или по ИД рейса (flight_id)
7. bookings: 9 706 657 записей, основные запросы будут по дате бронирования. Можно секционировать по дате бронирования (book_date).

## Выбор типа и диапазона

Поле book_date имеет тип timestamptz, секционировать лучше по диапазону. 
Если сделать секции по годам, то получится всего 3 секции с количеством записей больше 1,5 млн:

```sql
select date_trunc('year', book_date), count(*)
from bookings
group by 1;
```

```
"date_trunc"	"count"
"2027-01-01 00:00:00+03"	3192885
"2026-01-01 00:00:00+03"	4811461
"2025-01-01 00:00:00+03"	1702311
```

Если разделить по месяцам, то получится 25 секций с количеством записей около 400 тыс. 

Выбираю разделить на секции по диапазонам размером в месяц.

## Создание секционированной таблицы

Создаю схему и новую таблицу, идентичную bookings в ней - bookings_copy.bookings. 

С помощью скрипта создаю секции формата bookings_copy.bookings_<year>_<month>.  И секцию bookings_copy.bookings_other для значений, не входящих ни в один диапазон. 

Также скриптом с помощью цикла переношу данные из исходной таблицы в новую. После выполнения каждого шага - COMMIT, чтобы не нагружать систему. 

Для проверки, что все данные правильно распределены по секциям, нахожу минимальную, максимальную даты и количество записей в каждой секции.

## Оптимизация запросов
Запросы по ключевому столбцу с условием, затрагивающим небольшое количество секций, выполняются быстрее на секционированной таблице, чем на монолитной. 

Запросы по неключевому столбцу выполняются медленнее на секционированной таблице, так как используется Sec Scan по всем секциям. 

## Тестирование
При добавлении новых записей они сразу попадают в соответствующую секцию. Если секция не найдена, то новая запись попадает в секцию  bookings_copy.bookings_other.

При изменении даты бронирования так, что она становится неверной для своей секции, запись переносится в соответствующую новому значению секцию. 

Операция удаления работает корректно. 

## Вывод
Данный способ секционирования можно использовать, если запросы к таблице бронирования будут включать данные за один или несколько месяцев. В этом случае запросы будут выполняться быстрее.
